# REV-037 — SPEC-014, technical-architecture lens

**VERDICT: Reject.** The direction (agent lifecycle off n8n, mirroring SPEC-013) is right and honors #8, but two load-bearing mechanisms are architecturally unsound as written: durable suspend/resume is incompatible with the in-memory Anthropic loop it claims to persist, and the runtime's new dependence on the domain Postgres breaks the clean HTTP boundary the agents runtime has today. pg_cron cannot do the job assigned to it, and the wake source sits outside the SPEC-010 seam while contradicting its own "swappable axis" framing. Re-scope §4.2–§4.4 and return.

- **[TA1] Blocker** — Durable resume is incoherent with `runAnthropic`. The loop (`agents.ts:492–518`) carries the full `Anthropic.MessageParam[]` transcript in memory — assistant `tool_use` blocks paired to `tool_result` blocks by `tool_use_id`. §4.2's `working_memory` persists only "variables + a rolling conversation/**summary**"; a summary cannot reconstruct that transcript, so "resume into the right step with working memory intact" (§6) is not achievable — resume becomes a fresh, lossy re-prompt. Worse, the Anthropic API requires *every* `tool_use` in an assistant turn to be answered by a `tool_result` in the next user turn; if the model emits `wait_for`/`request_approval` alongside other tool calls in one turn (the common case), suspending there leaves the sibling `tool_use`s unanswered → a malformed `messages` array and a 400 on resume. §4.3 never addresses mid-turn suspend. FIX: persist the *full* `messages` transcript (not a summary) as the resume unit; specify that a wait tool must be the sole tool call in its turn (enforce in the behaviour prompt + a validator), or buffer sibling tool results before suspending; state the token/PII cost of transcript-in-`jsonb`.

- **[TA2] Blocker** — `agent_state`/`_events` give the agents runtime a Postgres coupling it does not have, and no owner. Today the generated agents runtime is HTTP-only to the spine (`tools.ts` `fetch(SPINE_URL)` + bearer; `package.json` deps are `@anthropic-ai/sdk`/`openai`/`express` — **no `pg`, no `DATABASE_URL`**, `agents.ts:663`). "Every `runAgent` invocation loads-or-creates a row; every step persists" (§4.2) forces a direct pg connection into `generated-agents`: a new driver dep, a new `DATABASE_URL` secret, and writes to a sibling table in the spine's own database — bypassing the spine's command API, auth gate, and `validate.ts`. Separately, *no one is assigned the DDL*: `postgres/schema.sql` is emitted per-aggregate by `postgresAdapter`; `migratePostgres` diffs domain aggregates only; the underscore tables + the `AFTER INSERT`→`pg_notify` trigger belong to none of them. So the unconditional `_events` INSERT §4.1 adds to `emit()` (`spine.ts:392`) will hit a non-existent table at runtime. FIX: decide the boundary explicitly — either agent state is written *through* new spine routes (keeps the HTTP boundary, reuses auth), or the coupling is accepted and `generated-agents` gains a documented pg dep; assign `_events`/`agent_state`/trigger DDL to a named generator and thread it through `migratePostgres`.

- **[TA3] Major** — pg_cron cannot re-invoke a Node agent loop. §4.4/D-timer says "pg_cron scanning `agent_state.wake_at` … re-invoking due runs." pg_cron executes SQL *inside* Postgres — it cannot make an HTTP call or run an LLM tool loop; at most it can `pg_notify` or flip a flag, after which the Node runtime must still do the work. That reduces the timer to a Node-side poller (`SELECT … wake_at <= now() FOR UPDATE SKIP LOCKED`), which the spec already has for event-wake — so pg_cron adds a superuser-only extension dependency (absent on much managed Postgres, per §7) for *zero* capability. Also unstated: `LISTEN` needs a dedicated long-lived connection, not the checked-in/out `pg.Pool` (`spine.ts:364`). FIX: drop pg_cron; make the timer a Node-side durable `SKIP LOCKED` poll over `wake_at`, same worker as event-wake; specify the dedicated `LISTEN` connection.

- **[TA4] Major** — The wake source is a parallel mechanism outside the SPEC-010 seam, and D6 contradicts the spec's own axis framing. Goals promise "one **swappable axis** (mirrors SPEC-013's `execution` axis)," but D6 then declares the wake source **auto-picked, not a binding dimension** — the opposite of SPEC-012/013's user-swappable axes. Architecturally, "wake" is a `react`/`sequence` concern that belongs in the `TechCapability` taxonomy + fidelity matrix and the `EngineAdapter` registry (SPEC-010 §4.1–4.3), not a bespoke `postgres|in-process|n8n` `WakeSource` enum dispatched off "deployment shape." As written it fragments the engine story: a contributor cannot add a wake backend via `registerEngine`. FIX: model wake as an engine capability resolved through the SPEC-010 binding/registry (n8n already provides `react`); if it must stay auto-derived, justify why it is exempt from the swappable-axis pattern and reconcile the Goals wording with D6.

- **[TA5] Major** — "No double-fire via target-kind routing" is under-modelled, and the `_events` write breaks byte-identity. `emit()` POSTs `on/<name>` to n8n **unconditionally for every event** when `N8N_BASE_URL` is set (`spine.ts:394`), and §4.1's `AFTER INSERT` trigger `pg_notify`s every event too — routing is not "each reaction to exactly one engine," it is *both fired, harmless only if no agent happens to wait on a workflow event* (unenforced). The decidability claim also assumes a *policy* can target an agent, but policies map `event→command` (`p.then` is a command id; `targets.ts:537`, `app.ts:199`); agent wake is the agent's own `wait_for`, so there is no codegen-time "target kind" on the policy to route by. Finally, the unconditional INSERT §4.1 adds to `emit()` changes the emitted spine bytes on **every** export, in tension with "byte-identical when no agent/trigger exists" (§Goals, §6). FIX: gate the `_events` write + table + trigger on "an agent/trigger exists" to preserve byte-identity; state precisely how an event routes to workflow-vs-agent when *both* a bound n8n `reaction_*` and a `waiting_event` agent key on the same event; correct the policy-vs-trigger conflation.

- **[TA6] Major** — The SPEC-009 determinism boundary (§4.6) is prose, not an enforceable seam. AL4 flags "a process marked determinism-critical routed to agent-mode," but no such authored field exists on `WorkflowInput` (which carries `mode: workflow|agent|external`), so AL4 cannot fire and "the SPEC-009 router is the law" is unbacked. §4.7's idempotency rests on the spine's `ON CONFLICT … DO UPDATE` upsert — but that only covers *create* commands; non-create routes are `UPDATE` (`spine.ts:485`), and a re-woken agent re-issuing a `send`/`delete` is bounded only by the SPEC-013 gate, which is *off* for `autonomous:true` grants. FIX: add the authored `determinismCritical` (or reuse a criticality field) so AL4 is decidable; scope the idempotency claim to create-commands and name the residual for update/send/delete explicitly.

- **[TA7] Minor** — `request_approval` resume (D7) specifies only the *outbound* half. "Reuses SPEC-013's `notify` channel" sends the question out; the human's reply must re-enter the runtime, match a `resume_token`, and resume the run — an *inbound* ingress + parse that D5's `POST /webhook/<path>` spine route implies but §4.3 never wires (how does an email/Slack reply become a `resume_token` POST?). FIX: specify the inbound resume path — the ingress route, how `resume_token` is presented and validated, and how a stale/replayed token is rejected.
